# 2030 Turnover Forecasting Methodology: Implementation Specification

## 1. Forecasting objective

The forecasting framework takes the completed turnover dataset produced by the turnover-estimation pipeline and predicts annual turnover for each eligible UK space company up to 2030.

The forecasting stage is separate from the missing-turnover model.

The first model produces:
[
\widehat{Y}_{i,0},
]
where (\widehat{Y}_{i,0}) is the estimated current or latest turnover for company (i).

The forecasting model then produces:
[
\widehat{Y}_{i,t+1}
f_m
\left(
Y_{i,t},
\mathbf{H}{i,t},
\mathbf{X}{i,t}
\right),
]
where:

* (m) denotes the company mission;
* (Y_{i,t}) is turnover at year (t);
* (\mathbf{H}_{i,t}) contains historical turnover features;
* (\mathbf{X}_{i,t}) contains company and financial predictors available at year (t).

A separate forecast model is trained for:

* Autonomous and Connected Earth;
* Beyond Earth;
* Resilient Earth.

The final model for each mission is selected through temporal backtesting.

## 2. Required input datasets

The forecasting framework requires two input tables.

### 2.1 Completed company baseline

One row per company.

Required fields:

```text
company_id
company_name
mission
baseline_year
baseline_turnover
turnover_source
baseline_lower
baseline_upper
baseline_reliability

```

`turnover_source` must distinguish between:

```text
observed
estimated

```

### 2.2 Historical company-year panel

One row per company per accounting year.

Required fields:

```text
company_id
company_name
mission
accounting_year
turnover
employees
total_assets
current_assets
liabilities
funding_raised
export_revenue
company_age
company_size
value_stream

```

Additional variables may be retained where they are consistently available.

The historical table must contain only values associated with the relevant accounting year. Current company values must not be attached retrospectively to earlier years unless their historical validity is known.

## 3. Data validation

Before forecasting samples are constructed, the following checks are performed.

### 3.1 Company identifier checks

Each row must contain a stable `company_id`.

The preferred identifier is the Companies House company number.

Rows without a valid identifier are retained in an exception file and excluded from model training until a stable identifier is assigned.

### 3.2 Duplicate checks

The combination:

```text
company_id + accounting_year

```

must be unique.

Where duplicate company-year rows exist:

1. retain the most reliable source;
2. otherwise retain the most recently updated record;
3. otherwise flag the conflict for manual review.

### 3.3 Turnover checks

Turnover must be:

```text
numeric
finite
greater than or equal to zero

```

Negative or non-numeric turnover values are converted to missing and logged.

### 3.4 Year checks

`accounting_year` must be a four-digit integer.

Rows outside the supported historical range are removed or reviewed.

### 3.5 Mission checks

Mission must be one of:

```text
ACE
Beyond Earth
Resilient Earth
Cross-Cutting

```

Cross-cutting companies are retained in the final output but excluded from primary model fitting unless a separate cross-cutting forecast model is later developed.

## 4. Historical panel construction

The historical financial data are reshaped into one ordered time series per company.

For each company:

```python
company_history = (
    panel
    .sort_values(["company_id", "accounting_year"])
)

```

The following company-history properties are calculated:

```text
n_turnover_years
first_turnover_year
latest_turnover_year
history_span
consecutive_history_length
years_since_latest_turnover

```

A company may contribute several training rows, but only where a valid future target exists.

## 5. Training-sample construction

The principal supervised-learning task is one-year-ahead forecasting.

For every company-year observation at year (t), the target is turnover in year (t+1):
[
Y^{\mathrm{target}}_{i,t}
Y_{i,t+1}.
]

A training row is created only where:

```text
turnover_t is available
turnover_t_plus_1 is available
company_id is valid
mission is valid

```

The target is generated by shifting turnover backwards within each company:

```python
panel["target_turnover_next_year"] = (
    panel
    .groupby("company_id")["turnover"]
    .shift(-1)
)

```

The corresponding target year is:

```python
panel["target_year"] = (
    panel
    .groupby("company_id")["accounting_year"]
    .shift(-1)
)

```

Rows are retained only where:

```python
target_year == accounting_year + 1

```

This prevents two non-consecutive accounting observations from being treated as a one-year transition.

A separate dataset may later be created for irregular intervals, but it must not be combined silently with the one-year training sample.

## 6. Historical evidence groups

Each company is assigned one evidence group.

```text
Group A: three or more observed turnover years
Group B: two observed turnover years
Group C: one observed turnover year
Group D: estimated baseline and no observed turnover history

```

The grouping variable is:

```text
forecast_evidence_group

```

The evidence group is used for:

* reliability reporting;
* subgroup validation;
* uncertainty estimation;
* model interpretation.

It is not automatically used as a growth predictor unless validation shows that it improves forecast performance without introducing bias.

## 7. Feature engineering

Feature engineering is performed independently for each forecast-origin row.

Only information available at or before year (t) may be used.

### 7.1 Turnover lag features

Create:

```text
turnover_t
turnover_lag_1
turnover_lag_2
turnover_lag_3

```

Implementation:

```python
group = panel.groupby("company_id")

panel["turnover_lag_1"] = group["turnover"].shift(1)
panel["turnover_lag_2"] = group["turnover"].shift(2)
panel["turnover_lag_3"] = group["turnover"].shift(3)

```

### 7.2 Log-turnover features

Create:

```text
log_turnover_t
log_turnover_lag_1
log_turnover_lag_2

```

using:
[
\log(1+Y).
]

Implementation:

```python
panel["log_turnover_t"] = np.log1p(panel["turnover"])

```

All values supplied to `log1p` must first be checked to ensure that they are non-negative.

### 7.3 Growth features

Annual log growth is:
[
g_{i,t}
\log(1+Y_{i,t})
\log(1+Y_{i,t-1}).
]

Create:

```text
log_growth_1y
log_growth_2y_mean
log_growth_3y_mean
growth_volatility
growth_acceleration
positive_growth_count
negative_growth_count

```

Implementation example:

```python
panel["log_growth_1y"] = (
    panel["log_turnover_t"]
    - panel["log_turnover_lag_1"]
)

```

### 7.4 Rolling turnover features

Create historical rolling statistics using only past and current observations:

```text
rolling_turnover_mean_2
rolling_turnover_mean_3
rolling_turnover_median_3
rolling_growth_mean_3
rolling_growth_std_3
historical_turnover_max
historical_turnover_min

```

The current row may be included because it is available at the forecast origin. Future rows must not be included.

### 7.5 History-quality features

Create:

```text
n_prior_turnover_values
consecutive_history_length
history_span_years
years_since_previous_turnover
has_growth_history
has_three_year_history

```

### 7.6 Company features

Retain available company characteristics:

```text
company_age
employees
total_assets
current_assets
liabilities
funding_raised
export_revenue
company_size
value_stream
mission

```

### 7.7 Derived financial features

Where valid denominator values exist, create:

```text

liabilities_to_assets
current_assets_to_assets
funding_per_employee
export_share
employee_growth

```

Ratios are set to missing where the denominator is zero or unavailable.

```latex
\subsubsection{7.8 Data Quality Features}
\label{subsubsec:forecast_data_quality_features}

In addition to financial and operational variables, a series of data-quality
features are generated to describe the reliability of the information
available for each company. These variables do not directly represent company
performance but instead provide the forecasting model with information about
the completeness and quality of the available evidence.

Potential data-quality features include:

\begin{itemize}
    \item baseline source (observed or estimated);
    \item number of historical turnover observations;
    \item years since the latest reported turnover;
    \item proportion of missing predictor variables;
    \item accounting-period length;
    \item availability of employee information;
    \item availability of financial statements;
    \item out-of-distribution indicators generated during the estimation
    framework.
\end{itemize}

Where appropriate, binary missingness indicators are created alongside
continuous variables so that the forecasting model can distinguish between a
true numerical value and an imputed observation.

All data-quality variables are calculated before preprocessing and are treated
as ordinary predictor variables during model development.

%=============================================================
\subsection{7.9 Forecast Feature Matrix}
\label{subsec:forecast_feature_matrix}
%=============================================================

Following feature engineering, all candidate variables are combined into a
single feature matrix.

Each row represents one company at one forecast origin, while each column
represents either:

\begin{itemize}
    \item turnover history;
    \item historical growth;
    \item company characteristics;
    \item financial variables;
    \item commercial variables; or
    \item data-quality indicators.
\end{itemize}

The response variable is the observed turnover in the following accounting
period.

For company \(i\) at forecast origin \(t\), the resulting supervised learning
problem becomes

\begin{equation}
\widehat{Y}_{i,t+1}
=
f(\mathbf{X}_{i,t}),
\end{equation}

where \(\mathbf{X}_{i,t}\) denotes the complete engineered feature vector.

Companies without an observed future turnover remain outside the supervised
training population and are introduced only after the final forecasting model
has been selected.

%=============================================================
\subsection{7.10 Forecast Preprocessing}
\label{subsec:forecast_preprocessing}
%=============================================================

Before model fitting, all predictor variables undergo preprocessing within a
machine-learning pipeline.

The preprocessing sequence is performed independently inside every validation
fold to prevent information leakage.

The preprocessing pipeline consists of the following stages.

\subsubsection{Step 1: Missing Value Imputation}

Numerical variables containing missing observations are imputed using the
median value calculated from the corresponding training partition.

Categorical variables are imputed using either the most frequent category or a
dedicated ``Unknown'' category where appropriate.

Imputation parameters are never estimated using validation or future
observations.

\subsubsection{Step 2: Categorical Encoding}

Categorical variables are converted into numerical representations suitable
for machine-learning algorithms.

Depending on the selected forecasting model, one of the following encoding
methods is applied:

\begin{itemize}
    \item one-hot encoding;
    \item ordinal encoding;
    \item frequency encoding;
    \item native categorical handling for CatBoost.
\end{itemize}

The chosen encoding strategy forms part of the model pipeline and is learned
only from the training data.

\subsubsection{Step 3: Numerical Scaling}

Algorithms that are sensitive to feature magnitude, such as Ridge Regression,
Lasso Regression and Support Vector Regression, receive scaled predictor
variables.

Standard scaling is applied as

\[
z
=
\frac{x-\mu}{\sigma},
\]

where the mean and standard deviation are calculated using only the training
partition.

Tree-based models do not require scaling and therefore bypass this stage.

\subsubsection{Step 4: Logarithmic Transformation}

Strongly skewed numerical predictors may be transformed using

\[
x'
=
\log(1+x),
\]

to reduce skewness and improve numerical stability.

The transformation is applied only where exploratory analysis demonstrates a
substantial departure from symmetry.

\subsubsection{Step 5: Feature Selection}

Optional feature-selection procedures are applied after preprocessing.

Candidate approaches include

\begin{itemize}
    \item correlation filtering;
    \item variance thresholding;
    \item recursive feature elimination;
    \item embedded feature selection;
    \item permutation importance.
\end{itemize}

Feature selection is performed entirely within the training partition and is
repeated independently for every validation split.

The final selected feature set is stored alongside the trained forecasting
model.

%=============================================================
\subsection{7.11 Forecast Model Bake-Off}
\label{subsec:forecast_bakeoff}
%=============================================================

Multiple forecasting algorithms are evaluated independently for each mission.

The bake-off follows the sequence:

\begin{enumerate}
    \item construct the historical company-year dataset;
    \item generate engineered features;
    \item preprocess the predictors;
    \item fit every candidate forecasting model;
    \item optimise hyperparameters;
    \item evaluate using rolling-origin validation;
    \item record performance metrics;
    \item compare against benchmark models;
    \item select the strongest forecasting model.
\end{enumerate}

Each mission therefore receives an independent forecasting model.

The candidate forecasting algorithms comprise:

\begin{itemize}
    \item persistence (no-growth);
    \item mission-average growth;
    \item company historical CAGR;
    \item linear regression;
    \item Ridge regression;
    \item Lasso regression;
    \item Elastic Net;
    \item Random Forest;
    \item Extra Trees;
    \item Gradient Boosting;
    \item Histogram Gradient Boosting;
    \item CatBoost (where available);
    \item Support Vector Regression.
\end{itemize}

Every candidate model is evaluated using the identical preprocessing,
validation and performance metrics to ensure a fair comparison.

The strongest-performing model is selected separately for ACE, Beyond Earth
and Resilient Earth before proceeding to the recursive forecasting stage.
```
